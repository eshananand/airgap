Air-Gapped by Design

Every command is a self-contained markdown file. No network calls, no plugin infrastructure, no data exfiltration. Your code, your context, your machine. Copy two directories to deploy anywhere.

Core Workflow

From idea to shipped code — structured, reviewed, and verified

💡
/design
Brainstorming → design → spec workflow. Guides a structured conversation from raw idea to approved design spec with clarifying questions, trade-off analysis, and iterative section approval.
Hard gate: no code before approval → /threat-model → /plan
📋
/plan
Takes an approved spec and produces a detailed implementation plan with atomic, bite-sized tasks. Includes file structure mapping, exact code, test commands, and dependency ordering.
→ /execute or /implement
▶
/execute
Inline plan execution with review gates. Executes each step sequentially, marking progress, summarizing at checkpoints, and surfacing blockers. Simpler alternative to /implement.
→ /verify → /finish
🤖
/implement
Subagent-driven plan execution — the recommended method. Dispatches a fresh subagent per task with two-stage review: spec compliance first, then code quality. Inlines all prompt templates.
→ /review → /finish
🔍
/review
Manually triggers the code-reviewer agent for quality review. Identifies scope, provides context about what was built and which plan it implements, then iterates until clean.
max 3 iterations → /finish
📥
/receive-review
Technical rigor when handling incoming review feedback. READ → UNDERSTAND → VERIFY → EVALUATE → RESPOND → IMPLEMENT. No performative agreement — every suggestion gets evaluated.
Hard gate: no blind implementation
🏁
/finish
Structured branch completion. Presents exactly 4 options with trade-offs: merge locally, open a PR, keep as-is, or discard (with typed confirmation). Cleans up worktrees.
4 completion options

Safety & Security

Scan, model, review, and audit — before vulnerabilities ship

🔒
/scan-secrets
Scans your codebase for leaked credentials before they reach git. Detects API keys, tokens, passwords, private keys, and connection strings across all major cloud providers and services.
Hard gate: blocks on definite secrets → /preflight
🛡
/threat-model
STRIDE-based threat modeling. Maps assets, entry points, and trust boundaries, then systematically analyzes Spoofing, Tampering, Repudiation, Information Disclosure, DoS, and Elevation of Privilege.
likelihood × impact matrix /design → here → /plan
🚨
/security-review
OWASP Top 10 focused security code review. Hunts for injection, XSS, broken auth, access control issues, and more. Distinct from /review which checks code quality.
CWE references severity ratings
✈
/preflight
Pre-deployment go/no-go gate. Orchestrates /verify, /scan-secrets, and /audit-deps into a single safety checklist. Checks for breaking changes, rollback plans, and environment configs.
Hard gate: NO-GO blocks deploy → /finish
📝
/postmortem
Structured, blameless incident analysis. Builds timelines, uses 5-Whys for root cause, separates contributing factors, and produces action items with owners, deadlines, and priorities.
blameless culture action items → /plan
📦
/audit-deps
Dependency vulnerability and license audit. Detects package managers automatically, checks for known CVE patterns, typosquatting indicators, and license incompatibilities.
multi-ecosystem → /preflight

Development Discipline

Guardrails that enforce engineering rigor at every step

🐛
/debug
Systematic 4-phase debugging: Root Cause Investigation, Pattern Analysis, Hypothesis Testing, Implementation. Forces root-cause analysis before any fix attempt. Questions architecture at 3+ failures.
Hard gate: no fix before root cause
🧪
/test
TDD red-green-refactor discipline. No production code without a failing test first. Includes rationalization prevention table, red flags list, and verification checklist.
Hard gate: test before code → /verify
✅
/verify
Evidence-based verification. No completion claims without fresh verification evidence. Runs tests, linters, type checkers, and confirms all acceptance criteria are met.
Hard gate: evidence before claims
🌳
/worktree
Creates isolated git worktrees for feature work. Smart directory selection, safety verification, and auto-detects project setup. Never creates worktrees on main/master.
git isolation → [work] → /finish
⚡
/parallel
Decomposes independent tasks into parallel subagents. Only parallelizes truly independent work — sequential dependencies stay sequential. Collects and integrates results.
independence required

Meta

Extend Airgap with your own commands

🔧
/new-skill
Creates new custom command files following Airgap conventions. Drafts the command, tests it with a subagent against a synthetic scenario, then writes and indexes it.
self-contained output README index

Agent

The shared review engine dispatched by multiple commands

🤖
code-reviewer (agent)
General-purpose review agent dispatched by /review, /design, /plan, /implement, and /security-review. Context passed at dispatch time determines what it reviews: spec, plan, code quality, or security.
4 review modes APPROVED or ISSUES FOUND

Command Chaining

/debug → fix → /verify → /review → /security-review → /finish
/test → /verify → /review → /finish
/scan-secrets → /preflight → /finish
/audit-deps → /preflight → /finish
/postmortem → action items → /plan
/worktree → [feature work] → /finish

Italic commands are optional but recommended in the chain.
